# PhoneShell 多设备架构重设计方案

## Context

当前 PhoneShell 架构是固定的 relay server 模式：一台 Linux 预设为中继服务器，手机和其他 PC 连上去。存在以下问题：
1. 用户部署时不知道哪台是 relay server
2. 不支持渐进式设备管理（先连一台，再逐步添加）
3. 手机端逻辑混乱（auth.request 仅页面级监听、单 WS 连接限制）
4. 设备互联没有授权机制

本方案将架构改为 **渐进式模式**：独立 → 单连 → 群组，手机作为管理中心。

---

## 1. 架构概述

### 1.1 设备三种模式

```
独立模式 (Standalone)     单连模式 (Single)        群组模式 (Group)
┌──────────┐            ┌──────┐               ┌──────────────┐
│  Device   │            │Phone │──WS──→Device  │    Relay      │
│  (QR待扫) │            └──────┘               │   Server      │
└──────────┘                                    │  (Device A)   │
                                                ├───────────────┤
                                                │ ← Phone (WS)  │
                                                │ ← Device B    │
                                                │ ← Device C    │
                                                │ ← Web Panel   │
                                                └───────────────┘
```

- **独立模式**：设备启动后运行本地 HTTP/WS 服务器，显示 QR 码，等待手机扫码
- **单连模式**：手机扫码后直接 WS 连接到设备，一对一控制终端
- **群组模式**：手机指定某台设备为 relay server，邀请其他设备加入群组

### 1.2 模式切换规则

| 从 | 到 | 触发方式 | 关键约束 |
|----|-----|---------|---------|
| 独立 → 单连 | 手机扫设备 QR | 手机与设备建立 WS 连接 |
| 单连 → 群组(relay) | 手机指定该设备为 relay | 设备 WS 服务器保持运行，启用中继功能，**手机连接不断开** |
| 单连 → 群组(client) | 收到邀请码 | 设备连接到 relay server |
| 群组 → 独立 | 群组解散/被踢出 | 断开 relay 连接，恢复本地 WS 服务器 |

**核心约束**：服务端/客户端切换时，手机的 WS 连接不得断开。设备从独立/单连升级为 relay server 时，在原有 WS 服务上增加中继功能即可。

### 1.3 邀请流程（简化版）

```
Phone                    Device A (relay)           Device B (standalone)
  │                          │                           │
  │  1. relay.designate ──→  │                           │
  │  ←── relay.designated    │                           │
  │                          │                           │
  │  2. invite.create ────→  │                           │
  │  ←── inviteCode          │                           │
  │                          │                           │
  │  3. HTTP POST /api/invite ──────────────────────────→│
  │     {relayUrl, inviteCode}                           │
  │                          │                           │
  │                          │  ←── group.join(inviteCode)│
  │                          │  ──→ group.join.response   │
  │                          │                           │
  │  ←── device.list (更新)  │                           │
```

步骤：
1. 手机告诉 Device A 成为 relay server
2. 手机请求 Device A 生成一次性邀请码（5 分钟过期）
3. 手机通过 HTTP POST 将邀请信息发送到 Device B（从 QR 码获取 Device B 的 HTTP 地址）
4. Device B 用邀请码连接到 Device A (relay)
5. 所有设备列表自动更新

### 1.4 权限模型

| 操作 | 谁发起 | 是否需要手机审批 |
|------|--------|-----------------|
| 手机连接设备 | 手机 | 否（手机是 admin） |
| 网页面板登录 | 网页 | 需要 QR 扫码 |
| 设备间互联（terminal.open.remote） | 设备/面板 | **需要 auth.request 审批** |
| 踢出设备 | 手机 | 否 |
| 解散群组 | 手机 | 否 |
| 修改设备名称 | 手机 | 否（广播同步） |

## 2. 协议变更

### 2.1 新增消息类型

| 消息类型 | 方向 | 说明 |
|---------|------|------|
| `relay.designate` | Phone → Device | 指定设备为 relay server |
| `relay.designated` | Device → Phone | 确认已成为 relay |
| `invite.create.request` | Phone → Relay | 请求生成邀请码 |
| `invite.create.response` | Relay → Phone | 返回 {inviteCode, relayUrl}，5 分钟过期 |
| `device.settings.update` | Phone → Relay | 修改设备设置 {deviceId, displayName, ...} |
| `device.settings.updated` | Relay → All | 广播设备设置变更 |
| `device.kick` | Phone → Relay | 踢出设备 {deviceId} |
| `device.kicked` | Relay → Target | 通知被踢设备 |
| `group.dissolve` | Phone → Relay | 解散群组 |
| `group.dissolved` | Relay → All | 通知所有设备群组已解散 |
| `panel.disconnected` | Relay → Phone | Web 面板断开通知 |

### 2.2 修改的消息类型

| 消息类型 | 变更 |
|---------|------|
| `group.join.request` | 新增 `inviteCode` 字段（与 groupSecret 二选一） |
| `device.register` | 新增 `mode` 字段: `standalone` / `relay` / `client` |
| `auth.request` | 新增 action: `terminal.open.remote`（设备间互联） |

### 2.3 邀请码 HTTP 端点

每台独立模式的设备暴露：
- `POST /api/invite` — 接收 `{relayUrl, inviteCode, groupId}`，设备自动连接 relay

## 3. 手机端变更 (HarmonyOS)

### 3.1 连接管理重构

**当前问题**：`WebSocketService.ets` 是单例单连接，无法同时连多台设备。

**方案**：重构为 `ConnectionManager`

```
ConnectionManager
├── singleConnections: Map<deviceId, DeviceConnection>  // 单连模式
├── groupConnection: DeviceConnection | null             // 群组模式
├── authRequestQueue: AuthRequest[]                      // 全局 auth 队列
└── onDeviceListChanged: callback                        // 统一设备列表更新
```

**文件变更**：

| 文件 | 操作 | 说明 |
|------|------|------|
| `services/WebSocketService.ets` | **重写** → `services/ConnectionManager.ets` | 多连接管理 |
| 新建 `services/DeviceConnection.ets` | 新增 | 单个 WS 连接封装（原 WebSocketService 核心逻辑下沉） |
| 新建 `services/AuthManager.ets` | 新增 | 全局 auth.request 处理，待处理队列，弹窗触发 |
| `services/PreferencesUtil.ets` | 修改 | 新增：保存单连设备列表、群组 relay 地址 |

`DeviceConnection` 封装：
- 一个 WS 连接 + deviceId + mode
- connect / disconnect / send / onMessage
- 自动重连 + 心跳

`ConnectionManager` 职责：
- `connectSingle(url, deviceId)` — 建立单连
- `designateRelay(deviceId)` — 将单连设备升级为 relay
- `inviteToGroup(deviceUrl, inviteCode)` — HTTP POST 邀请
- `disconnectDevice(deviceId)` — 断开单连
- `dissolveGroup()` — 解散群组
- 统一暴露 `allDevices: DeviceInfo[]`（合并单连 + 群组设备列表）

### 3.2 Auth 全局处理

**当前问题**：`auth.request` 只在 `DeviceListPage` 监听，切换页面后丢失。

**方案**：`AuthManager` 全局单例
- 在 `ConnectionManager` 内注册消息监听
- 收到 `auth.request` → 加入 `pendingQueue`
- 任何页面可调用 `AuthManager.showPending()` 弹出半屏弹窗
- 使用 `@ohos.promptAction` 或自定义 `CustomDialogController` 在当前页面顶层显示
- 不再用 `router.pushUrl` 跳转到 `AuthorizationPage`（改为弹窗）

## 4. Linux 服务端变更 (Node.js)

### 4.1 新增 RelayClient 模块

当前 linux2 只有 relay-server 模式，缺少 client 模式。需要新增 RelayClient 用于：被邀请加入群组后，连接到其他设备的 relay server。

| 文件 | 操作 | 说明 |
|------|------|------|
| `src/relay/relay-client.ts` | **新建** | WS 客户端，自动重连，group.join 支持 inviteCode |
| `src/relay/mode-manager.ts` | **新建** | 模式状态机：standalone → relay / client，切换逻辑 |

`relay-client.ts` 参考 C# 版 `RelayClient.cs`：
- 连接到 relay server URL
- 发送 `device.register` + `group.join.request`（带 inviteCode）
- 转发终端 I/O 到 relay
- 自动重连

`mode-manager.ts` 管理设备模式切换：
```
standalone ──(relay.designate)──→ relay
standalone ──(invite received)──→ client
relay ──────(group.dissolve)───→ standalone
client ─────(kicked/dissolve)──→ standalone
```

### 4.2 邀请系统

| 文件 | 操作 | 说明 |
|------|------|------|
| `src/relay/relay-server.ts` | 修改 | 新增 invite.create / group.join(inviteCode) 处理 |
| `src/auth/invite-manager.ts` | **新建** | 邀请码生成、存储、过期、一次性验证 |
| `src/server/app.ts` | 修改 | 新增 `POST /api/invite` 端点 |

邀请码规则：
- 8 位随机字母数字，`crypto.randomBytes` 生成
- 5 分钟 TTL，一次性使用
- 内存存储（Map），使用后立即删除
- `group.join.request` 新增 `inviteCode` 字段验证

### 4.3 设备设置广播

`relay-server.ts` 新增处理：
- `device.settings.update` → 更新 group.json → 广播 `device.settings.updated`
- `device.kick` → 断开目标设备 WS → 从 group.json 移除 → 广播 device.list
- `group.dissolve` → 断开所有设备 → 清理 group.json → 广播 `group.dissolved`
- Web panel 断开 → 发送 `panel.disconnected` 给手机

### 4.4 模式切换不断连实现

当设备收到 `relay.designate`：
1. 当前已在运行 HTTP/WS 服务器（手机已连接）
2. 初始化 relay-server 的路由逻辑（消息转发、群组管理）
3. 创建 group.json
4. 回复 `relay.designated` {relayUrl: 当前服务器 URL}
5. 手机的 WS 连接全程保持

**实现关键**：relay 功能是"叠加"在现有 WS 服务上的，不是重启服务器。

## 9. 验证方案

1. **独立模式**：Linux 启动 → 显示 QR → 手机扫码 → 建立单连（待加入设备）→ 终端可用
2. **多设备待加入**：手机扫第二台设备 → 两台设备都在待加入设备区域 → 各自终端独立可用
3. **创建群组**：手机在待加入设备长按设备 → 移入群组 → 选择 relay → 群组建立
4. **邀请加入**：手机扫第三台设备 → 自动邀请加入群组 → 设备列表更新
5. **设备互联 auth**：Web 面板请求打开远程设备终端 → 手机弹出 auth 弹窗 → 审批
6. **设备设置**：手机重命名设备 → 所有端同步更新显示名
7. **踢出设备**：手机踢出某设备 → 设备返回独立模式 → 群组列表更新
8. **群组解散**：手机解散群组 → 所有设备返回独立模式
9. **模式切换不断连**：整个过程手机 WS 无断开/重连
10. **Web 面板关闭通知**：关闭网页 → 手机收到通知
11. **Bug1 保持**：手机断开 → 网页不被踢回 QR 页面
