# PhoneShell Linux 部署重要说明

本文档记录实际部署中遇到的关键问题和解决方案，适用于所有 Linux 云服务器部署。

---

## 0. 端口架构说明

典型的生产部署涉及**两个端口**，各自职责不同：

```
手机/浏览器
    │
    ▼
┌──────────────────────────┐
│  Nginx（外部端口 9090）   │  ← 对外暴露，客户端实际连接的端口
│  - TLS 终结（可选）       │
│  - Basic Auth 保护        │
│  - WebSocket 升级代理     │
└──────────┬───────────────┘
           │ proxy_pass
           ▼
┌──────────────────────────┐
│ PhoneShell（内部端口 19090）│  ← 仅监听内网，不对外暴露
│  - Relay WebSocket 服务   │
│  - Web Panel 页面         │
│  - REST API               │
└──────────────────────────┘
```

| 端口 | 监听地址 | 用途 | 是否对外 |
|------|----------|------|----------|
| **9090**（Nginx） | `0.0.0.0:9090` | 反向代理入口，提供 Basic Auth、限流、TLS 等安全层 | 是，需要在防火墙和云安全组放行 |
| **19090**（PhoneShell） | `10.x.x.x:19090` / `127.0.0.1:19090` | PhoneShell 后端实际服务端口 | 否，仅 Nginx 内部访问 |

### 为什么需要两个端口？

- **安全隔离**：PhoneShell 本身不提供 Basic Auth 和限流，通过 Nginx 代理可以增加认证和防护层
- **灵活部署**：可以在 Nginx 层配置 TLS（`wss://`），不需要改动 PhoneShell 代码
- **端口分离**：PhoneShell 监听内网地址，即使防火墙配置失误，外部也无法直连后端

### 如果不需要 Nginx？

可以让 PhoneShell 直接监听对外端口，省去 Nginx：

```json
{
  "publicHost": "你的外网IP",
  "port": 9090
}
```

此时只有一个端口，直接放行即可。但生产环境**推荐使用 Nginx**。

## 1. QRCoder 依赖缺失（构建问题）

### 现象

服务启动后，日志持续报错：

```
[relay-server] Accept error: Could not load file or assembly 'QRCoder, Version=1.4.3.0, Culture=neutral, PublicKeyToken=...' The system cannot find the file specified.
```

每当有客户端连接时就会触发，导致 Web Panel 的 QR 码无法生成。

### 原因

`PhoneShell.Core.dll` 内部引用了 `QRCoder`（用于生成二维码），但 `PhoneShell.Headless.csproj` 没有显式声明这个依赖。使用 `PublishSingleFile=true` 发布时，只有项目直接引用的 NuGet 包才会被打包进单文件，`PhoneShell.Core.dll` 的间接依赖不会自动包含。

### 解决方案

在 `PhoneShell.Headless.csproj` 中显式添加 QRCoder 引用：

```xml
<ItemGroup>
  <PackageReference Include="QRCoder" Version="1.4.3" />
</ItemGroup>
```

然后重新发布：

```bash
dotnet publish PhoneShell.Headless/PhoneShell.Headless.csproj \
  -c Release -r linux-x64 --self-contained true \
  -p:PublishSingleFile=true -o publish/linux-x64
```

## 2. NAT / 反向代理环境下 QR 码地址错误（关键）

### 现象

Web Panel 页面的二维码中，`serverUrl` 显示为内网地址：

```
ws://10.x.x.x:19090/ws/
```

手机扫码后无法连接服务器。

### 原因

`PhoneShell.Core.dll` 通过枚举本机网卡自动检测可达地址。在云服务器上，外网 IP 是通过 NAT 映射的，网卡上只有内网 IP，程序无法感知外网地址。

如果前面还有 Nginx 反向代理（例如 Nginx 监听 9090，代理到后端 19090），那么端口也会不匹配。

### 解决方案

在 `config.json` 中配置 `publicHost` 字段：

```json
{
  "publicHost": "你的外网IP:外部端口",
  "port": 19090
}
```

#### 场景 A：直接暴露端口（无反向代理）

外网 IP 为 `1.2.3.4`，PhoneShell 直接监听 9090 端口：

```json
{
  "publicHost": "1.2.3.4",
  "port": 9090
}
```

QR 码将显示 `ws://1.2.3.4:9090/ws/`

#### 场景 B：通过 Nginx 反向代理（推荐）

Nginx 监听外部端口 9090，反向代理到 PhoneShell 的 19090：

```json
{
  "publicHost": "1.2.3.4:9090",
  "port": 19090
}
```

QR 码将显示 `ws://1.2.3.4:9090/ws/`

#### 场景 C：使用域名 + TLS

```json
{
  "publicHost": "relay.example.com:443",
  "port": 19090
}
```

也支持 CLI 参数和环境变量：

```bash
# CLI
--public-host 1.2.3.4:9090

# 环境变量
PHONESHELL_PUBLIC_HOST=1.2.3.4:9090
```

## 3. 防火墙端口放行

### 常见遗漏

只放行了 Nginx 端口（如 9090），忘记放行 PhoneShell 直连端口，或者反过来。

### 检查清单

```bash
# 查看服务监听端口
ss -tlnp | grep -E "9090|19090"

# 查看 ufw 防火墙规则
ufw status

# 放行端口
ufw allow 9090/tcp    # Nginx 对外端口
ufw allow 19090/tcp   # 如需直连 PhoneShell（通常不需要）
```

**重要**：除了本机防火墙（ufw/iptables），云服务商还有**安全组**规则，需要在云控制台（腾讯云/阿里云/AWS 等）同步放行对应端口的入站规则。

## 4. GroupSecret 必须修改

### 问题

配置文件中默认的 `relayAuthToken` 是占位符 `replace-with-a-long-random-secret`，不修改会导致安全风险。

### 解决方案

生成随机密钥并写入配置：

```bash
# 生成 48 字符的随机 hex 字符串
openssl rand -hex 24
```

写入 `config.json`：

```json
{
  "relayAuthToken": "生成的随机字符串"
}
```

同步更新环境变量文件（如使用 systemd）：

```
PHONESHELL_RELAY_TOKEN=生成的随机字符串
```

## 5. Nginx WebSocket 代理配置

如果使用 Nginx 反向代理，必须正确配置 WebSocket 升级，否则手机客户端无法建立 WebSocket 连接。

关键配置项（`/etc/nginx/sites-available/phoneshell`）：

```nginx
server {
    listen 9090;
    server_name _;

    # WebSocket 路径 — 必须配置 Upgrade 头
    location /ws/ {
        proxy_pass http://127.0.0.1:19090;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 86400s;
        proxy_send_timeout 86400s;
    }

    # Web Panel
    location /panel {
        proxy_pass http://127.0.0.1:19090;
    }

    location /panel/ {
        proxy_pass http://127.0.0.1:19090;
    }

    # Panel API
    location /api/ {
        proxy_pass http://127.0.0.1:19090;
    }
}
```

## 快速部署检查清单

部署完成后，按顺序确认以下各项：

- [ ] `dotnet publish` 构建成功，无缺失依赖错误
- [ ] `config.json` 中 `publicHost` 已设置为 `外网IP:Nginx端口`
- [ ] `config.json` 中 `relayAuthToken` 已替换为随机密钥
- [ ] PhoneShell 服务正常运行：`systemctl status phoneshell`
- [ ] 日志无 `QRCoder` 或其他 assembly 错误：`journalctl -u phoneshell -f`
- [ ] 本机防火墙已放行端口：`ufw status`
- [ ] 云安全组已放行端口
- [ ] Nginx 配置正确并已 reload：`nginx -t && systemctl reload nginx`
- [ ] 健康检查通过：`curl http://127.0.0.1:19090/ws/healthz`
- [ ] Web Panel 可访问：浏览器打开 `http://外网IP:9090/panel`
- [ ] QR 码显示正确的外网地址
- [ ] 手机扫码可成功加入群组
